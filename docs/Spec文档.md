# Lumi-Draw 开发规格文档 (Spec)

## 1. 开发原则

### 1.1 最小原则
- 从基础框架搭建开始
- 每个任务完成一个一级界面
- 然后逐个完成二级界面
- 每次开发前参考原型图确保还原度
- 完成后进行自主验证和测试

### 1.2 开发流程
```
1. 参考原型图 -> 开发界面
2. 对比原型图 -> 检查功能完整性 + 样式还原度
3. 修复差异 -> 确保无漏做/多做/还原度低
4. Git commit -> 提交代码
5. 部署公网 -> 浏览器验证
6. 确认通过 -> 继续下一任务
```

### 1.3 质量保障
- 每完成一个界面，对比原型图逐元素检查
- 检查是否存在漏做的功能模块
- 检查是否存在多做的非需求功能
- 检查样式还原度（间距/字号/颜色/布局）
- 防止项目逐步偏离和代码质量劣化

## 3. 每个任务的标准流程

### 3.1 开发前
- [ ] 阅读原型图对应页面的HTML结构
- [ ] 确认需要实现的功能点清单
- [ ] 确认依赖的API接口

### 3.2 开发中
- [ ] 前端页面开发（参考原型图一比一复刻）
- [ ] 后端API开发（如有新增）
- [ ] 数据对接

### 3.3 开发后
- [ ] 对比原型图逐元素检查功能完整性
- [ ] 检查样式还原度（布局/间距/字号/颜色/动效）
- [ ] 检查无漏做模块
- [ ] 检查无多做非需求功能
- [ ] Git commit
- [ ] 部署到公网
- [ ] 浏览器访问验证
- [ ] 确认通过后继续下一任务

## 4. 自主验证清单

### 4.1 功能验证
- 每个按钮/交互是否可点击且有响应
- 列表是否正确渲染数据
- 分页/加载更多是否正常
- 弹窗/抽屉是否正确打开和关闭
- Tab切换是否正确
- 表单输入和提交是否正常

### 4.2 样式验证
- 整体布局是否与原型一致
- 间距/字号/颜色是否还原
- 卡片/按钮/标签等组件样式是否正确
- 空状态/加载状态是否处理
- 深色模式适配（如需要）

### 4.3 数据验证
- API请求是否正确发送
- 响应数据是否正确解析
- 错误处理是否完善
- Loading/骨架屏是否正确显示

## 5. 技术约定

### 5.1 前端
- 组件命名使用 PascalCase
- 文件命名使用 kebab-case
- 页面目录与 pages.json 配置一致
- 公共组件放 components/
- 工具函数放 utils/
- 状态管理放 store/

### 5.2 后端
- Controller 只处理请求/响应
- Service 处理业务逻辑
- Entity 定义数据结构
- DTO 定义请求参数
- 使用 class-validator 做参数校验
- 使用 TypeORM 做数据库操作

### 5.3 通用
- 所有API返回统一格式 `{code, msg, data}`
- 所有错误通过全局异常过滤器处理
- 所有列表接口支持分页
- 所有时间字段使用 ISO 8601 格式
- 所有ID使用 UUID
